# 领域知识约束增强的多智能体强化学习调度方法

## **一、研究背景与问题动机**

### **1.1 行业痛点**

流程工业（炼化、塑纤等）生产过程具有 **工序链条长、调度环节多、运行扰动频繁** 的特征。调度方案的优劣直接影响：

- 系统产量与产品保供能力
- 能耗水平与切换成本
- 中间库存稳定性
- 动态扰动下的恢复速度

### **1.2 现有方法的局限**

| **方法类型** | **优势** | **不足** |
| --- | --- | --- |
| **传统优化方法**（MILP、启发式搜索、滚动优化） | 能较好刻画工艺约束与操作边界 | 依赖强模型假设，在线重调度能力有限，动态适应性弱 |
| **纯强化学习方法** | 数据驱动的自适应决策能力强 | 领域知识仅停留在奖励工程层面，容易产生不可执行决策 |
| **现有 MARL 调度方法** | 支持多智能体并行决策 | 缺少显式知识约束机制，跨工序协调性能不足 |

### **1.3 核心矛盾**

> **纯数据驱动** → 决策可行性差、可解释性弱  
**纯规则/优化** → 动态适应性差、在线性能不足  
**两者如何融合？** → 这正是本文要解决的问题
>

## **二、现有研究空白分析**

本工作重点针对以下两个核心空白和一个方法评估空白展开：

### **核心空白 1：领域知识注入深度不足**

- 现有 RL/MARL 调度方法中，领域知识的使用方式普遍较浅：
    - 多数仅将知识用于 **奖励惩罚项设计**（如违规扣分）
    - 或以 **简单规则预筛选** 的方式做离线过滤
- 这些方式**未深入改变策略学习本身的决策结构**——智能体的状态表达、搜索空间和学习信号并未因领域知识而发生本质变化
- **本文的切入**：将知识注入到 **状态编码 → 动作可行域过滤 → 策略偏置 → 协同奖励** 四个层面，使知识 **显式改变 RL 的决策结构**，而非仅作为外挂规则

### **核心空白 2：弱耦合多工序协同建模缺失**

- 流程工业大量装置并非强管道刚性耦合，而是通过 **中间池和产品池形成弱耦合库存联动**
- 现有研究的典型处理方式：
    - 单智能体集中式：难以扩展到多装置场景
    - 独立多智能体：各自局部最优，容易导致 **上下游物料失配、全局库存失衡**
    - 已有 MARL 协同方法：多面向车间调度或 Job-Shop 等离散制造场景，**缺少针对流程工业弱耦合库存联动特性的协同建模**
- **本文的切入**：通过 CTDE 框架 + 全局协调信号 + 协同奖励，实现"局部自治 + 全局协调"的统一

### **方法评估空白：缺少扰动驱动的自适应调度评估**

- 很多调度方法仅在 **静态或单一工况** 下验证性能
- 缺少在 **需求激增、装置降额、库存突降等多类型动态扰动** 下的系统性评估
- 导致方法的 **鲁棒性和工程适用性** 难以得到充分验证
- **本文的切入**：构建 static / default / severe 三类渐进扰动场景，系统评估方法的协同能力和恢复能力

## **三、研究目标与核心科学问题**

### **3.1 核心科学问题**

> **在多工序、分布式、弱耦合的流程工业场景中，如何利用领域知识约束多智能体强化学习的决策空间与协同机制，使调度策略在动态扰动下同时具备可行性、协调性和自适应性。**
> 

### **3.2 三个研究子问题**

| **编号** | **子问题** | **对应创新点** |
| --- | --- | --- |
| Q1 | 工艺规范、设备能力、物料守恒和专家经验 **如何进入 MARL 决策过程**（而非只作离线规则） | 知识约束增强机制 |
| Q2 | 多工序智能体如何在局部自治的同时 **保持上下游物料与能量协同** | 协同机制设计 |
| Q3 | 扰动发生时，调度策略如何 **快速在线重构** 并维持约束可行性 | 自适应调度机制 |

### **3.3 论文贡献（拟定三条）**

1. 提出一种 **领域知识约束增强的多智能体强化学习调度框架 (KC-CTDE)**，将工艺知识显式融入状态建模、动作约束和奖励设计
2. 设计一种 **面向物料衔接与全局协调的多智能体协同机制**，提升弱耦合多工序场景中的全流程调度一致性
3. 构建一种 **动态扰动下的自适应调度评估框架**，验证方法在可行性、鲁棒性和综合绩效上的优势

### **4.2 马尔可夫决策过程建模**

本文将炼厂多装置协同调度问题建模为一个 **知识约束下的分散式部分可观测马尔可夫决策过程（KC-Dec-POMDP）**，形式化定义如下：

### **4.2.1 智能体集合**

```
I = {CDU1, CDU2, FCC1, FCC2, ROHU, DHC, DFHC}，共 N=7 个智能体
```

每个智能体对应一套物理装置，具有独立的局部观测和动作空间。

### **4.2.2 全局状态空间 S**

全局状态 `s^t` 在每个时步 `t` 由以下三部分组成：

| **状态分量** | **符号** | **维度** | **具体内容** |
| --- | --- | --- | --- |
| **装置状态** | `U^t` | 7 x 5 | 每个装置的 {负荷 load, 运行模式 mode, 是否可用 available, 降额系数 derate_factor, 上次切换时步 last_switch_step} |
| **库存状态** | `M^t` | 9 | 9 个主库存节点的当前液位 level（crude_pool, naphtha_feed_pool, ..., lpg_pool） |
| **需求状态** | `D^t` | 3 | 汽油、柴油、LPG 的当前需求值 |

```
s^t = (U^t, M^t, D^t)
```

状态空间为连续空间，维度 = 7*5 + 9 + 3 = **47 维**。

### **4.2.3 局部观测空间 O_i**

每个智能体 `i` 仅能观测到与自身相关的局部信息（分散执行约束）：

| **观测维度** | **内容** | **来源** |
| --- | --- | --- |
| 1 | 自身当前负荷 `load` | 装置状态 |
| 2 | 运行模式 `mode_is_normal` | 装置状态 |
| 3 | 是否可用 `available` | 装置状态 |
| 4 | 降额系数 `derate_factor` | 装置状态 |
| 5 | 上游库存 `upstream_inventory` | 上游罐区液位 |
| 6 | 下游库存 `downstream_inventory` | 下游罐区液位 |
| 7 | LPG 库存 `lpg_inventory` | 公共产品池 |
| 8 | 汽油当前需求 `gasoline_demand` | 全局需求信号 |
| 9 | 柴油当前需求 `diesel_demand` | 全局需求信号 |
| 10 | LPG 当前需求 `lpg_demand` | 全局需求信号 |

```
o_i^t = (load_i, mode_i, avail_i, derate_i, inv_up_i, inv_down_i, inv_lpg, d_gas, d_dsl, d_lpg)
```

每个智能体的观测为 **10 维连续向量**，经知识增强状态编码后映射为 **10~14 维离散元组**（详见模块 2）。

### **4.2.4 动作空间 A_i**

每个智能体的动作为一个离散联合决策：

| **动作维度** | **取值** | **候选数** |
| --- | --- | --- |
| 负荷档位 `load` | {0.0, 0.6, 0.8, 1.0} | 4 |
| 运行模式 `mode` | {normal, energy_saving} | 2 |
| LPG 外销（仅 FCC1） | {false, true} | 2 |

```
a_i^t = (load, mode, export_lpg)
```

- 普通装置：4 * 2 = **8 个候选动作**
- FCC1（LPG 外销代理）：4 * 2 * 2 = **16 个候选动作**
- 联合动作空间规模：8^6 * 16 = **4,194,304**

`0.0` 表示停机/保底运行，`0.6/0.8/1.0` 分别对应低/中/高三档加工负荷。

**知识约束后**：通过 Action Filter 过滤不可行动作，实际每步每智能体可行动作数约 3-8 个。

### **4.2.5 状态转移函数 P**

状态转移 `s^{t+1} = P(s^t, a^t, xi^t)` 由三部分驱动：

**A. 确定性转移（动作执行）**

```
对每个装置 i，执行动作 a_i^t:

  1. 消耗上游库存:
     inv_upstream -= load_i * feed_rate_i

  2. 产出到下游库存（按固定产率）:
     对每个输出池 j:  inv_j += load_i * feed_rate_i * yield_{i->j}

  3. 更新装置内部状态:
     U_i.load = load_i,  U_i.mode = mode_i

  4. 消耗产品池满足需求:
     served_p = min(inv_p, demand_p),  inv_p -= served_p
     unmet_p = demand_p - served_p

  5. LPG 外销（若 FCC1 触发）:
     exported = min(inv_lpg - buffer, export_cap)   若 inv_lpg > target
```

**B. 随机性转移（需求波动）**

```
demand_p^{t+1} = max(floor_p, base_p + amplitude_p * pattern(t+1) + shift_p)
```

其中 `pattern(t)` 为确定性6步周期波动，`shift_p` 为扰动事件累积偏移。

**C. 外生扰动转移**

```
xi^t 包含三类扰动事件:
  - demand_surge:  shift_p += magnitude    （需求激增）
  - derate:        derate_factor_i = magnitude  （装置降额）
  - inventory_drop: inv_tank -= magnitude   （库存骤降）
```

扰动在预设时步触发，模拟真实炼厂运行中的突发工况变化。

### **4.2.6 奖励函数 R**

全局即时奖励 `R(s^t, a^t)` 由 7 项加权组成：

```
R = R_throughput + R_demand - C_energy - C_switch - C_violation - C_inventory + R_coordination
```

各项详细定义：

| **分量** | **公式** | **设计意图** |
| --- | --- | --- |
| **吞吐收益** R_throughput | `0.9 * sum(load_i * feed_rate_i)` | 鼓励有效加工 |
| **需求满足** R_demand | `3.2*(served_gas+served_dsl) + 1.7*served_lpg + 0.8*exported_lpg - 4.6*(unmet_gas+unmet_dsl) - 2.8*unmet_lpg` | 保供优先，缺货严厉惩罚 |
| **协同库存** R_coordination | 每装置: `max(0, 5.5 - 1.6*abs(level-target)/target)` | 全局库存趋近目标 |
| **能耗成本** C_energy | `sum(load_i * energy_cost_i)` | 控制运行能耗 |
| **切换成本** C_switch | `switch_count * 2.5` | 抑制频繁换挡 |
| **违规成本** C_violation | `violations * 25.0` | 零容忍硬约束违反 |
| **库存偏差** C_inventory | `10.0*sum(abs(level-target))/sum(target) + 3.0*boundary_penalty` | 库存平衡 + 边界保护 |

### **4.2.7 知识约束扩展**

与标准 Dec-POMDP 的核心区别在于，本文引入领域知识 `K = {K^h, K^s, K^c}` 对 MDP 的三个要素进行约束：

| **MDP 要素** | **标准 Dec-POMDP** | **KC-Dec-POMDP（本文）** |
| --- | --- | --- |
| **观测** | 原始 `o_i^t` | 知识增强编码 `z_i^t = Enc(o_i^t, K_i)` |
| **动作空间** | 全候选集 `A_i` | 过滤后可行集 `A_i^feas(s^t, K^h) ⊆ A_i` |
| **奖励** | 纯数据驱动奖励 | 知识偏置 + 协同信号增强奖励 |

### **4.2.8 优化目标**

在知识约束条件下，求解最优联合策略使折扣累积奖励最大化：

```
max_{pi} J(pi) = E[ sum_{t=0}^{T-1} gamma^t * R(s^t, a^t) ]

s.t.  a_i^t in A_i^feas(s^t, K^h)     对所有 i, t
      T = 24 (单日滚动调度)
      gamma = 0.96 (折扣因子)
```

### **模块 1：领域知识形式化与结构化表示**

将领域知识分为三类，每类都有明确的数据结构：

| **类型** | **符号** | **内容** | **作用方式** |
| --- | --- | --- | --- |
| **硬约束知识** | `hard_limits` | 设备最小/最大负荷、降额系数 | 动作过滤（硬裁剪） |
| **软约束知识** | `soft_preferences` | 偏好负荷区间、切换成本、能耗差异 | 策略偏置 + 奖励塑形 |
| **协同知识** | `coupling_targets` | 各装置下游池区的目标库存 | 协同奖励 + 全局信号 |

**实现细节**：系统启动时，从案例配置中自动构建 `KnowledgeSnapshot`：

```
对每个装置 i:
  hard_limits[i] = { min_load, max_load }
  soft_preferences[i] = { preferred_low, preferred_high, switch_cost }
  coupling_targets[i] = { target_downstream_inventory }
```

例如 `CDU1` 的硬约束为 `min_load=0.6, max_load=1.0`，软偏好区间为 `[0.8, 1.0]`，切换惩罚系数为 `2.0`。这些知识不仅用于事后评估，而是 **在决策前直接参与动作筛选和策略偏置计算**。

**软约束惩罚函数**（影响奖励塑形）：

```
soft_penalty(i, a) =
  + 2.0           若 load 不在 [preferred_low, preferred_high]
  + switch_cost   若当前模式与动作模式不同（发生切换）
  + energy_normal 或 energy_saving   根据选择的运行模式
```

**基础状态编码**（不含知识特征时，10 维离散元组）：

| **维度** | **原始观测** | **分桶方式** | **桶数** |
| --- | --- | --- | --- |
| 1 | 当前负荷 load | 阈值 `(0.7, 0.9)` | 3 |
| 2 | 运行模式 | normal=1 / energy_saving=0 | 2 |
| 3 | 是否可用 | available=1 / 0 | 2 |
| 4 | 降额系数 | 阈值 `(0.8, 0.95)` | 3 |
| 5 | 上游库存/目标 | 阈值 `(0.35, 0.8, 1.2, 1.8)` | 5 |
| 6 | 下游库存/目标 | 阈值 `(0.35, 0.8, 1.15, 1.6)` | 5 |
| 7 | LPG库存/目标 | 阈值 `(0.35, 0.8, 1.15, 1.6)` | 5 |
| 8 | 汽油需求/基准 | 阈值 `(0.8, 1.05, 1.3)` | 4 |
| 9 | 柴油需求/基准 | 阈值 `(0.8, 1.05, 1.3)` | 4 |
| 10 | LPG需求/基准 | 阈值 `(0.75, 1.1, 1.45)` | 4 |

**关键设计**：所有库存和需求维度都采用 **相对值**（`level / target_level`、`demand / base_demand`），而非绝对值。这样不同装置、不同规模的案例共享同一套分桶语义。

**知识增强特征**（开启后额外追加 4 维）：

| **维度** | **含义** | **计算方式** |
| --- | --- | --- |
| 11 | 偏好负荷区域 | 0=低于偏好下界, 1=偏好区间内, 2=高于偏好上界 |
| 12 | 上游库存风险 | 5级: min以下=0, 低于0.8*target=1, 目标附近=2, 接近max=3, 超max=4 |
| 13 | 下游库存风险 | 同上，5 级风险等级 |
| 14 | 需求偏向 | 0=柴油主导, 1=均衡, 2=汽油主导, 3=LPG高需求 |

**这 4 维正是知识注入的核心**：它们将工艺领域中"什么是安全的""什么是紧缺的"编码为状态的一部分，使同样的物理状态在不同工况下产生 **不同的状态键**，从而驱动策略学到差异化的响应。

### **模块 3：动作可行域过滤 (Action Filter)**

过滤器在每个时步、每个智能体决策前执行，按 **四层串联规则** 逐一检查候选动作：

```
对每个候选动作 a:

  Rule 1 - 可用性检查:
    若装置不可用且 load > 0 则过滤掉

  Rule 2 - 负荷边界检查:
    有效上限 = max_load * derate_factor
    若 load < min_load 或 load > 有效上限 则过滤掉

  Rule 3 - 上游原料充足性检查:
    所需进料 = load * feed_rate
    若 upstream_tank.level < 所需进料 则过滤掉

  Rule 4 - 下游溢出检查:
    对每个非 sink 的输出池:
      预测库存 = 当前库存 + 所需进料 * yield_factor
      若 预测库存 > max_level * 1.02 则过滤掉
```

**注意**：`load = 0.0`（停机/保底）始终作为兜底动作保留，确保在极端工况下智能体至少有一个可行选择。

**实际效果**：在工业案例中，7 个装置各有 8-16 个候选动作，经过滤后典型可行动作数为 3-8 个，策略搜索空间缩减 **50%-70%**。

### **模块 4：协同奖励与全局协调信号**

协同奖励设计分为两层：**全局奖励**（环境层面）和 **智能体局部奖励**（混合局部+全局信号）。

**A. 全局奖励分解（7 项）**

| **奖励分量** | **计算公式** | **系数说明** |
| --- | --- | --- |
| **吞吐收益** | `0.9 * throughput` | 鼓励有效加工 |
| **需求满足** | `3.2*(served_gas+served_dsl) + 1.7*served_lpg` | 汽柴油权重高于LPG |
| **需求缺口惩罚** | `-4.6*(unmet_gas+unmet_dsl) - 2.8*unmet_lpg` | 缺货惩罚远大于满足收益 |
| **协同库存收益** | 每装置: `max(0, 5.5 - 1.6*abs(level-target)/target)` | 越接近目标，收益越高 |
| **能耗成本** | `-energy_cost` | 直接扣减 |
| **切换成本** | `-switch_count * 2.5` | 每次换挡扣 2.5 |
| **违规成本** | `-violations * 25.0` | 严厉惩罚，确保零违规 |
| **库存偏差** | `-10.0*inv_dev/target - 3.0*boundary_penalty` | 偏离目标+触碰边界双重惩罚 |

**B. 智能体局部奖励（混合设计）**

每个智能体的奖励由 **局部项** 和 **全局协调项** 加权混合：

```
r_agent = 0.45 * r_local + 0.55 * (r_global / N) + r_coordination_bonus
```

其中局部项包括：

- `load * 7.5`（吞吐激励）
- 局部库存偏差惩罚 `|downstream - target| * 0.04`
- 偏好区间外惩罚 `2.0`
- 能耗成本 `mode_cost`
- 需求缺口分摊 `0.55*(unmet_gas+unmet_dsl) - 0.25*unmet_lpg`
- 动作覆写重罚 `20.0`（若提议动作被环境覆写为可行动作）

**C. 全局协调信号（Coordination Signal）**

集中 Critic 的全局状态键中嵌入 6 维协调信号：

| **信号** | **含义** | **分桶阈值** |
| --- | --- | --- |
| `inventory_pressure` | 全局库存偏压/目标总量 | `(0.25, 0.5, 0.8, 1.1)` |
| `product_shortage` | 产品总缺口/基准需求总量 | `(0.05, 0.25, 0.5, 0.9)` |
| `upstream_risk` | 原油池低于安全线的风险值 | `(1.0, 2.0, 4.0)` |
| `gasoline_cover` | 汽油覆盖率 | `(0.5, 1.0, 1.5)` |
| `diesel_cover` | 柴油覆盖率 | `(0.5, 1.0, 1.5)` |
| `lpg_cover` | LPG 覆盖率 | `(0.5, 1.0, 1.5)` |

**协调奖金**（Coordination Bonus）：根据协调信号对每个智能体动态附加奖励。例如当汽油覆盖率 <1.0 时，面向 gasoline_pool 的装置选择高负荷动作可获得 `+4.0` 奖金；反之覆盖率 >1.4 时低负荷可获得 `+1.5`。

### **模块 5：服务副头 (Service Head)**

服务副头是一个 **独立的 Actor-Critic 对**，与主 Actor-Critic 平行训练，但专注于 **保供目标**。

**A. 架构设计**

```
最终决策 = argmax [ P_main(a|s) + w_service * P_service(a|s) + bias_knowledge(a) ]
```

- `P_main`：主 Actor 的偏好值，学习综合收益
- `P_service`：服务副头的偏好值，专注保供
- `w_service`：动态权重，根据当前短缺程度自适应调整
- `bias_knowledge`：知识偏置项

**B. 服务副头奖励（独立于主奖励）**

```
r_service = 1.5*(served_gas + served_dsl) + 0.5*served_lpg
          - 2.0*(unmet_gas + unmet_dsl) - 0.8*unmet_lpg
          + 6.0 * load * multiplier   （当对应产品覆盖率 < 1.0 时）
          - 8.0 * multiplier          （若停机且产品短缺 > 8.0）
          - 10.0                      （若提议动作被覆写）
```

**C. 动态权重机制（核心设计）**

服务副头的权重 `w_service` 不是固定的，而是根据 **三个维度** 动态计算：

| **维度** | **实现逻辑** |
| --- | --- |
| **短缺程度** | shortage>28 或 cover<0.25 时权重x1.0; shortage>14 或 cover<0.6 时x0.8; shortage>4 或 cover<0.9 时x0.45; 否则不激活 |
| **场景严重度** | 根据扰动事件数、降额事件数、库存骤降数和需求波幅计算 severity_score: >=11时x1.0, >=8时x0.55, 其余x0.25 |
| **装置关联度** | 按装置是否直接产出短缺产品计算 multiplier: 如 FCC 在 gasoline 短缺时 multiplier=0.85, CDU 作为上游进料装置在双产品短缺时 multiplier=0.9 |

**设计意图**：

- 静态场景下服务副头几乎不激活，不干扰主策略的综合优化
- 中等扰动下适度介入，对关键装置施加保供偏好
- 强扰动下全力激活，防止策略在极端短缺下崩溃

**D. 知识偏置项（Knowledge Bias）**

在主+服务副头偏好值之上，叠加基于当前工况的确定性偏置：

| **工况条件** | **偏置规则** | **目的** |
| --- | --- | --- |
| 上游库存接近 min | 停机 +1.6, 低负荷 +0.9, 高负荷 -1.4 | 防止缺料 |
| 下游是汽油池且短缺 | 高负荷 +1.7, normal +0.5, 停机 -1.6 | 汽油保供 |
| 下游是柴油池且短缺 | 高负荷 +1.7, normal +0.5, 停机 -1.6 | 柴油保供 |
| 下游库存低于 0.75*target | 高负荷 +1.2, normal +0.4 | 补库 |
| 下游库存高于 1.2*target | 低负荷 +0.8, 节能 +0.5, 高负荷 -1.0 | 抑制过产 |
| 负荷在偏好区间内 | +0.25 | 鼓励稳态运行 |
| LPG高库存且开外销 | +0.9 | 鼓励适时外销 |
| LPG低库存且开外销 | -1.2 | 制止低库存外销 |

### **4.4 算法训练流程**

```
1. 初始化环境、7 个装置智能体、知识约束集合、经验池
2. 观测局部状态 s_i^t 和全局状态 s^t
3. 知识编码器 → 知识增强表示 z_i^t
4. Actor 根据 z_i^t 生成候选动作
5. Action Filter 依据硬约束裁剪不可行动作
6. 环境执行联合动作 a^t → 返回奖励和新状态
7. 协调模块计算全局协调信号，更新联合奖励
8. 轨迹存入经验池，CTDE 更新 actor 和 critic
9. 重复直至收敛
```

**训练超参数**：

| **参数** | **值** | **说明** |
| --- | --- | --- |
| episodes | 220 | 训练轮数 |
| actor_alpha | 0.08 | Actor 学习率 |
| critic_alpha | 0.12 | Critic 学习率 |
| gamma | 0.96 | 折扣因子 |
| temperature | 1.2 -> 0.25 | 探索温度线性退火 |
| service_head_weight | 1.4 | 服务副头基础权重 |
| service_reward_weight | 1.3 | 服务奖励权重（sweep 标定） |
| inventory_reward_weight | 0.4 | 库存协调权重（sweep 标定） |
| energy_reward_weight | 0.2 | 能耗权重（sweep 标定） |
